# DocGen 产品策略与规划

## 1. 背景

DocGen 当前已经具备可用的 MVP 能力：

- 通过 CLI 交互收集基本需求
- 生成文档大纲并等待用户确认
- 按章节顺序生成内容
- 导出完整 Markdown 文档
- 支持本地存储、断点恢复、基础质量审查

这说明产品已经不只是一个简单的 prompt 脚本，而是一个具备项目状态、生成流程和本地持久化能力的早期文档生产工具。

但从产品定位和使用体验来看，当前版本仍存在几个明显问题：

- 产品定位偏宽，容易显得“什么都能做一点”
- 大纲有时过长，包含不少非必要内容
- 生成前的交互更像表单收集，而不是真正的需求澄清
- 编辑闭环较弱，缺少局部重写和反馈回灌
- 来源能力不足，无法有效整合网页或外部平台内容
- 文档内容与样式偏平淡，学习型文档的阅读体验不够强

因此，下一阶段不应继续沿着“泛文档生成器”扩张，而应收敛到更明确、更适合当前形态的产品方向。

## 2. 产品定位

### 2.1 最终定位

DocGen 应聚焦为：

**面向开发者、技术作者和教育内容创作者的 AI 技术教程 / 学习指南 / 技术手册生成工具。**

### 2.2 核心价值

产品不只是“生成一篇文档”，而是帮助用户完成以下完整过程：

1. 明确学习目标和读者画像
2. 规划合适的教程结构
3. 生成可学习、可复习、可面试准备的内容
4. 在生成后继续进行局部修改和迭代
5. 输出更像真正教程的结构化成品

### 2.3 重点场景

优先支持以下三类文档：

- 技术教程
- 学习指南
- 操作手册 / 技术手册

短期内应弱化以下方向：

- 学术论文
- 泛知识百科式文档
- 过于宽泛的“任何文档都能生成”

这样可以让产品叙事更聚焦，也能让现有的大纲生成、章节生成、术语一致性等能力发挥更大的价值。

## 3. 当前主要问题

### 3.1 大纲控制不足

当前大纲生成容易追求“覆盖全面”，但教程类文档更需要：

- 结构清晰
- 学习路径明确
- 篇幅可控
- 内容有取舍

仅凭一个主题生成大纲，容易出现以下问题：

- 目录过长
- 层级过深
- 塞入大量边缘内容
- 缺少对读者目标和学习阶段的针对性

### 3.2 需求收集方式偏弱

当前交互更像填表式输入，虽然能得到主题、类型、受众等字段，但还不足以支撑高质量教程设计。

教程生成前应该明确：

- 这份文档是用于入门、系统学习、查漏补缺，还是面试准备
- 目标读者处于什么水平
- 希望控制在什么篇幅
- 更偏原理、实践，还是二者结合
- 是否包含路线图、案例、FAQ、面试题等模块

### 3.3 编辑闭环缺失

现有流程更偏向“一次生成”，不利于真实使用中的反复打磨。

现实中的写作流程通常是：

1. 先生成初稿
2. 阅读与审阅
3. 提出修改意见
4. 局部重写
5. 后续章节继续遵守新规则

如果没有局部重写和反馈回灌，用户很容易退回人工编辑，产品价值会明显下降。

### 3.4 来源能力不足

教程和学习指南往往依赖外部资料：

- 官方文档
- 博客文章
- GitHub README / docs
- 社区经验
- 平台内容，如公众号、专栏等

仅依赖用户输入主题和少量本地文件，会限制内容深度、时效性和可信度。

### 3.5 成品可读性不强

当前输出更像普通 Markdown 长文，缺乏学习资料常见的表达层次，例如：

- 重点提示
- 注意事项
- 常见错误
- 示例说明
- 面试提示
- 图示说明
- 小结与复盘

这会影响教程类内容的使用体验，也会降低文档的“成品感”。

## 4. 产品策略

下一阶段建议围绕 4 个能力模块推进。

### 4.1 需求澄清

目标：在生成大纲前先理解用户真正要什么。

建议把当前“字段收集”升级为“需求访谈”，采用两阶段流程：

#### 阶段 A：需求访谈

围绕以下问题进行多轮确认：

- 内容主要用于什么：入门、系统学习、查漏补缺、面试准备、项目实战
- 目标读者的基础水平
- 期望篇幅：速读版、标准版、深入版
- 偏原理还是偏实战
- 是否需要学习路线图
- 是否需要面试精选内容
- 是否需要项目案例、FAQ、术语速查等模块

#### 阶段 B：教程设计摘要

在正式生成大纲前，先输出一页结构化摘要供用户确认，包括：

- 目标读者
- 核心目标
- 文档范围
- 建议章节数
- 需要包含的模块
- 风格与深度建议

只有在用户确认后，再进入正式大纲生成。

### 4.2 大纲控制

目标：避免目录失控，让大纲更像教程设计，而不是知识堆叠。

建议新增以下控制维度：

- 章节数上限
- 最大层级深度
- 总篇幅目标
- 内容优先级选择
- 模块开关

推荐支持的模块包括：

- 学习路线图
- 核心概念讲解
- 实战案例
- 常见误区
- 面试高频题
- 小结 / 复盘
- 术语速查
- 延伸阅读

大纲生成逻辑应由以下因素共同决定，而不是仅由主题决定：

- 主题
- 用户意图
- 读者画像
- 篇幅限制
- 模块选择
- 内容侧重点

### 4.3 编辑闭环

目标：让 DocGen 从“一次性生成器”升级为“可持续打磨的写作工具”。

建议优先实现以下能力：

#### 局部重写

支持：

- 按章节重写
- 按小节重写
- 按目标重写，例如扩写、简化、改成更适合新手、补充例子、补充对比

重写时应自动带入：

- 原始文本
- 所在章节摘要
- 全文术语表
- 已有用户反馈

#### 反馈回灌

允许用户输入自然语言修改意见，例如：

- 第 3 章太抽象，多给例子
- 第 2 节默认读者有基础，不适合新手
- 统一把 function calling 改成工具调用

系统应将这些反馈转成结构化编辑规则，持久化保存，并在后续生成或重写时自动继承。

建议新增项目级数据：

- `editor_feedback.json`
- `revision_history/`
- `section_status`

### 4.4 内容增强与样式增强

目标：让产物更像高质量学习资料，而不只是平铺正文。

建议从两层推进：

#### 语义增强

在内容中引入明确的语义块：

- 重点
- 注意
- 常见错误
- 示例
- 最佳实践
- 面试提示
- 图示说明
- 小结

#### 模板样式

支持用户提供风格参考，例如：

- Markdown 示例文档
- DOCX 示例文档
- 已有教程样例

系统从中提取样式偏好，例如：

- 标题风格
- 提示框风格
- 表格与列表偏好
- 是否常用重点总结
- 是否偏好图例或练习题

这样可以让生成结果更贴近用户实际想要的成品风格。

## 5. 来源能力规划

### 5.1 短期策略

先做稳定、易控、价值高的来源：

- 普通网页
- 官方文档站
- 博客文章
- GitHub README / docs
- 用户上传资料

### 5.2 中期策略

逐步扩展到更复杂的平台内容：

- 微信公众号
- 知乎专栏
- 掘金
- B 站稿件或字幕

### 5.3 实施原则

来源能力应设计为“来源连接器”体系，而不是一开始承诺“全网抓取”。

建议支持两种模式：

- 用户提供 URL，系统抓取并清洗
- 用户上传已导出的内容文件，系统解析并纳入生成上下文

对于复杂平台，需要额外考虑：

- 访问限制
- 内容清洗质量
- 时效性
- 合规边界

## 6. 版本规划

### v0.2 聚焦与控纲

目标：收敛产品场景，减少第一次生成跑偏。

建议内容：

- 收敛产品定位到教程 / 学习指南 / 技术手册
- 重写 README、产品说明和 CLI 提示文案
- 增加需求澄清流程
- 增加教程设计摘要确认步骤
- 增加大纲长度和模块控制
- 优化默认教程大纲模板

### v0.3 编辑闭环

目标：让用户能够对初稿进行有效迭代。

建议内容：

- 增加 `rewrite` 命令
- 支持章节级重写
- 支持小节级重写
- 支持用户反馈持久化
- 支持重写时自动继承历史反馈规则

### v0.4 内容结构增强

目标：让输出更符合学习和备面试场景。

建议内容：

- 支持学习路线图模块
- 支持面试精选模块
- 支持常见误区、小结、术语速查等组件
- 支持偏入门、偏系统、偏面试的内容模式

### v0.5 来源与样式增强

目标：提升信息丰富度和成品表现力。

建议内容：

- 支持网页抓取与内容提取
- 优先支持官方文档、博客、GitHub 文档
- 评估并逐步支持公众号等复杂来源
- 支持样式模板输入或风格参考生成

## 7. 最终建议

DocGen 下一阶段最值得坚持的方向不是继续做“更泛的文档生成”，而是做“更像真实技术教程生产流程”的产品。

一句话总结为：

**DocGen 应成为一个会先理解学习目标、再规划教程结构、还能持续修改打磨的 AI 技术教程生成工具。**

围绕这个方向，短期最重要的不是增加更多文档类型，而是先做好以下四件事：

1. 收敛场景到教程 / 学习指南 / 手册
2. 在大纲前增加需求澄清和大纲控制
3. 建立局部重写和反馈回灌能力
4. 逐步增强来源整合和样式表达

当这些能力建立起来后，DocGen 才会从“一个能写文档的 AI 工具”真正进化成“一个能规划、生成、修改、组织教程的文档生产系统”。
